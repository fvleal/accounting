From RolesGuard behavior (established in Phase 04-01):
- Registered as global APP_GUARD after JwtAuthGuard
- Reads @Roles() metadata via Reflector
- If NO @Roles metadata on route -> returns true (pass-through)
- If @Roles metadata exists -> checks user.permissions array includes ALL required permissions
- Returns 403 ForbiddenException if permissions missing

From test/accounts.e2e-spec.ts (E2E test pattern):
- MockJwtAuthGuard reads `x-test-auth` header as JSON: `{ sub, email, permissions }`
- To test 403: send request with x-test-auth that has a different sub AND lacks the required permission
- Existing ownership tests follow pattern: create account with user A, then call endpoint with user B's sub
